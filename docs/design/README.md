# 设计资料总览

本目录用于集中放置项目设计资料。目标不是把所有草稿都包装成正式论文，而是让读者能快速看到：这个项目有需求分析、对象建模、数据库设计、接口分层和业务实现沉淀。

## 推荐展示顺序

1. [用例与业务流程](use-cases.md)：先说明系统服务哪些角色，以及动态、集市、悬赏、审核等关键业务如何闭环。
2. [分析类图](class-analysis.md)：展示每个业务场景中的视图、控制器、实体和核心关系。
3. [关系模型](relational-model.md)：展示用户、学校、内容、交易、通知等核心数据关系。
4. [数据库设计](database-design.md)：进一步落到 MySQL 表结构、字段、索引和外键。
5. [校园集市实现总结](market-implementation.md)：用一个完整模块说明前后端和数据库如何协同。

## 文件说明

| 文件 | 展示价值 |
| --- | --- |
| `use-cases.md` | 适合讲需求分析、角色划分、主流程和异常流程 |
| `class-analysis.md` | 适合讲领域建模和三层架构下的对象协作 |
| `relational-model.md` | 适合讲 ER 模型和核心业务关系 |
| `database-design.md` | 适合讲表结构设计、索引、外键和业务字段 |
| `database-diff-notes.md` | 适合讲数据库设计迭代过程 |
| `missing-table-design.md` | 适合讲后续扩展和需求补齐能力 |
| `backend-notes.md` | 适合讲后端分层、文件职责和接口落点 |
| `market-implementation.md` | 适合讲一个功能模块从页面到 API 再到数据表的实现 |

## 讲解提示

- 讲设计资料时，不需要逐行展开。先讲“为什么这么分层”，再选一个核心流程展开。
- 集市和悬赏是最适合展示业务闭环的两个模块：它们有发布、详情、交互、订单或进度状态。
- 数据库设计可以重点讲 `users`、`products`、`missions`、`orders`、`notifications` 等表，以及为什么用内容类型字段承载评论/点赞的复用关系。
- 归档目录中的开发记录可以作为“项目迭代痕迹”，但正式讲解优先使用本目录的整理版文档。
