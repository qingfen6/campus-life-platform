# 用例与业务流程

本文档是对原始 `用例设计/用例规约.md`、通信图和顺序图的整理版说明，用于快速说明系统的用户角色、核心业务闭环和异常处理思路。

## 角色划分

| 角色 | 主要职责 |
| --- | --- |
| 学生 | 发布动态、发布商品、购买商品、发布悬赏、接取悬赏、参加活动、互动评论和接收通知 |
| 社团管理人 | 发布社团公告、发起活动、审核加入社团人员、维护社团信息和成员 |
| 学校管理员 | 发布通知公告、发布校园活动、审核加入校园人员、编辑学校信息、调整社团管理员 |
| 系统管理员 | 发布系统公告、管理平台活动、审核学校申请、维护首页信息、调整学校管理员 |

## 核心业务闭环

### 校园动态

学生完成登录和学校身份认证后，可以发布图文动态。系统需要校验用户状态、保存动态内容、处理媒体资源，并在动态流中展示。动态后续可以被点赞、评论、分享或触发通知。

展示重点：
- 体现内容生产和互动闭环。
- 适合结合 `src/pages/Home.js`、`server/routes/homeRoutes.js` 和评论组件讲解。

### 校园集市

学生可以发布二手商品，买家可以浏览商品、查看详情、议价、创建订单并进行模拟支付。订单流程会影响商品状态，并向卖家发送通知。

展示重点：
- 体现商品、订单、账户、交易和通知之间的协作。
- 适合结合 [校园集市实现总结](market-implementation.md) 和订单/支付类图素材讲解。

### 悬赏任务

学生可以发布悬赏任务，系统检查账户余额并冻结悬赏金额。其他学生可以申请接单，发布者审核后任务进入执行流程，完成后进行验收和结算。

展示重点：
- 体现状态流转和金额冻结的业务设计。
- 适合说明为什么需要 `missions`、`mission_takes`、`accounts`、`transactions` 等表。

### 校园人员审核

学生申请加入学校后，学校管理员查看申请资料和证明材料，可以通过、拒绝或要求补充材料。审核结果会更新申请状态和用户认证状态，并发送通知。

展示重点：
- 体现管理后台的真实价值，而不是只做数据展示。
- 适合结合管理端 API 和学校管理员页面讲权限边界。

## 关键异常流程

| 场景 | 异常 | 处理方式 |
| --- | --- | --- |
| 发布动态/商品/悬赏 | 未完成学校身份认证 | 提示先完成认证，并引导到认证入口 |
| 商品购买 | 商品已售出 | 阻止下单并提示返回商品列表 |
| 订单/悬赏支付 | 账户余额不足 | 提示充值或调整金额 |
| 悬赏接取 | 任务已关闭 | 阻止接单并提示任务不可接取 |
| 人员审核 | 证明材料不完整 | 学校管理员请求补充材料，申请状态进入待补充 |
| 活动报名 | 报名截止或人数已满 | 阻止报名，并给出明确原因 |

## 讲解方式

可以用“一个核心流程 + 一个后台流程”进行讲解：先讲校园集市或悬赏任务，说明学生端如何完成业务闭环；再讲校园人员审核，说明管理后台如何承接平台治理。这样比逐个枚举功能更有效，也更能展示系统设计能力。
